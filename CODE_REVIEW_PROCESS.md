## 2. 审查前准备

### 2.1 开发者自查清单

提交代码前，请确认：

- [ ] **代码完整**：功能实现完整，无临时代码
- [ ] **自测通过**：本地运行和测试通过
- [ ] **构建成功**：`npm run build` 无错误
- [ ] **测试完整**：新增功能有相应测试
- [ ] **文档更新**：README、注释、JSDoc 更新
- [ ] **提交规范**：符合 Git 提交规范
- [ ] **代码整洁**：无 console.log、无注释掉的代码

### 2.2 PR/MR 描述模板

请在创建 Pull Request 或 Merge Request 时，使用以下模板：

```markdown
## 📝 描述

简要描述这次变更的目的和内容。

## 🔄 变更类型

- [ ] 新功能 (feat)
- [ ] Bug 修复 (fix)
- [ ] 文档更新 (docs)
- [ ] 代码重构 (refactor)
- [ ] 性能优化 (perf)
- [ ] 测试相关 (test)
- [ ] 构建/工具 (chore)

## 📋 变更内容

- 变更点1
- 变更点2
- 变更点3

## 🧪 测试覆盖

- [ ] 已添加单元测试
- [ ] 已手动测试
- [ ] 已验证所有功能正常

## 📖 文档

- [ ] 已更新 README
- [ ] 已添加 JSDoc 注释
- [ ] 已更新 API 文档

## 🎯 关联 Issue

Closes #123

## 📸 截图/演示

如有 UI 变更，添加截图或演示链接
```

## 3. 审查流程

### 3.1 阶段一：自动化检查

1. **类型检查**

```bash
npm run build
```

2. **测试运行**

```bash
npm run test:run
```

3. **代码规范**（如配置了 eslint/prettier）

### 3.2 阶段二：代码审查

#### 3.2.1 审查角色

- **小 PR**：<20 行变更：1 位审查
- **中 PR**：20-200 行变更：1-2 位审查
- **大 PR**：>200 行变更：2+ 位审查，建议拆分

#### 3.2.2 审查点

1. **通读代码**：理解整体变更
2. **使用标准**：对照 CODE_REVIEW_STANDARDS.md
3. **先看全局**：架构、设计、职责
4. **细节确认**：实现细节、边界条件
5. **提出问题**：记录疑问点
6. **提供方案**：给出改进建议

#### 3.2.3 审查时间

- **目标**：24-48 小时内完成
- **延期**：如无法及时审查，提前告知
- **优先级**：按 PR 紧急程度安排

## 4. 审查反馈指南

### 4.1 反馈分类

| 级别    | 颜色/符号   | 含义                 | 处理要求           |
| ------- | ----------- | -------------------- | ------------------ |
| 🔴 阻塞 | ❌ 必须修改 | 代码有严重问题       | 必须修改后才能合并 |
| 🟡 重要 | ⚠️ 建议修改 | 有重要问题，建议修改 | 建议修改           |
| 🟢 轻微 | 💡 可选改进 | 有改进空间           | 可选修改           |
| ✅ 赞赏 | 👍 赞赏     | 代码写得好！         |

---

### 4.2 审查反馈

- ：

1. **具体**：指出具体代码位置
2. **建议**：提供具体改进
3. **建设性**：尊重贡献者
4. **解释原因**：为什么需要
5. **举例**：如何改进

**好反馈模板**：

````markdown
### 🔴 必须修改

**文件**：`components/VideoCard.tsx:45-52

**问题**：这段代码有性能问题，每次渲染都创建新函数

**建议**：使用 useCallback 包装

```typescript
// ❌ 当前
const handleClick = () => {
  /* ... */
};

// ✅ 建议
const handleClick = useCallback(() => {
  /* ... */
}, [dependency]);
```
````

### 4.3 开发者回应指南

1. **感谢**：感谢审查反馈
2. **讨论**：有不同意见时，友好地
3. **解释**：不认同解释原因
4. **修改**：按反馈修改代码
5. **回复**：每个评论都回应

## 5. 合并标准

### 5.1 合并前检查清单

- [ ] 所有审查通过
- [ ] 所有评论已回应
- [ ] CI 自动化检查
- [ ] 无冲突
- [ ] 提交记录清晰

### 5.2 合并策略

- **小变更**：Squash Merge（压缩合并）
- **功能分支**：Merge Commit（保留提交历史）

## 6. 审查最佳实践

### 6.1 审查者技巧

1. **先肯定**：先找代码优点，再提改进
2. **问问题**：不懂就问，理解
3. **理解背景**：理解
4. **尊重**：尊重不同编码风格
5. **提供替代**：提供替代方案
6. **及时**：及时，不拖延

### 6.2 被审查者

1. **开放心态**：接受反馈
2. **积极回应**：及时回应
3. **解释决策**：有必要的决策
4. **感谢**：感谢帮助
5. **学习成长**：每次审查都是学习机会

### 6.3 常见审查点

| 常见问题           | 如何处理                   |
| ------------------ | -------------------------- |
| 代码风格不同       | 统一规范，求同存异         |
| 性能与可读性权衡   | 优先可读性，性能关键时优化 |
| 过度工程与架构争议 | 讨论，达成一致             |
| PR 太大            | 建议拆分 PR，分多次合并    |

## 7. 特殊情况处理

### 7.1 紧急修复

对于紧急 bug 修复：

- 简化流程，1 位审查
- 但仍需保证质量
- 事后补充完善

### 7.2 重构

分歧处理

- 保持理性讨论
- 寻找折方案
- 记录决策
- 后续复盘改进
